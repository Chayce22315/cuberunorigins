## High‑Level Architecture Overview

### State Machine
- `GameStateMachine` wraps a `GKStateMachine` and holds references to the current `SKScene`, audio, input, and score managers. It forwards `update(deltaTime:)` calls to the underlying state machine.
- Concrete states (`MainMenuState`, `GameplayState`, `ChaseState`, `PausedState`, `GameOverState`) each subclass `GKState` and implement `didEnter`, `update`, and `willExit` to set up UI, drive gameplay logic, and clean up.
- Transitions are performed via `GameStateMachine.enter(_: )` (e.g., from `GameplayState` to `ChaseState` when the score reaches `Constants.chaseScoreThreshold`).

### Entities & Components (ECS pattern)
- **Entity classes** (`PlayerEntity`, `EvilCubeEntity`) inherit from `GKEntity` and compose a set of components.
- **Components** live in `Components/`:
  - `RenderComponent` creates the visual `SKSpriteNode`.
  - `PhysicsComponent` attaches a `SKPhysicsBody` to the render node.
  - `MovementComponent` updates horizontal position each frame and provides a `jump()` method.
  - `AnimationComponent` runs idle and jump `SKAction`s.
  - `AudioComponent` plays short sound effects.
  - `AIComponent` (used by `EvilCubeEntity`) drives simple enemy behaviour, random state changes and occasional glitch triggers.
- Components access each other via the entity (`entity?.component(ofType:)`).

### Managers (Singleton services)
- `AudioManager` – background music & effect playback.
- `UIManager` – score label and placeholder pause / game‑over menus.
- `CameraManager` – smooth following camera with optional shake (`SKAction.shake`).
- `ObstacleManager` – spawns `SpikeEntity` obstacles at `Constants.obstacleSpawnInterval`.
- `GlitchManager` – singleton that can apply either an overlay fade or a screen‑shake to the current scene; used by `AIComponent`.

### Scene
- `GameScene` is the main `SKScene`. It configures physics, installs UI, starts the `GameStateMachine` with `MainMenuState`, and on each frame updates the state machine and all managers.
- When entering `GameplayState`, a `PlayerEntity` is instantiated, added to the scene, and the camera is set to follow the player.

### Constants
- Central `Constants.swift` defines colors, physics category bit masks, speeds, score thresholds, spawn intervals, and other magic numbers.
- Nested structs (`ZPosition`, `Enemy`) group related values.

### Testing
- Tests live in `Tests/` and use the standard XCTest target (`CubeRunOriginsTests`). Example tests verify obstacle spawn timing and that `GlitchManager` exercises both glitch types.

### Folder Layout (relevant high‑level groups)
- `Components/` – reusable behavior units attached to entities.
- `Entities/` – concrete `GKEntity` subclasses.
- `Managers/` – singleton services shared across the game.
- `Scenes/` – scene‑specific files (`GameScene`, cutscenes, event definitions).
- `Sources/` – additional source files that don’t fit the other categories (e.g., alternate AI implementation).
- `Tests/` – unit and logic tests.
- `Constants.swift` – global configuration.
- `GameStateMachine.swift` and state files – core game flow.

## Additional Notes
- The project uses **XcodeGen** (`project.yml`) to generate the Xcode workspace; the generated project lives in `cuberunoriginsmac/cuberunorigins.xcodeproj`.
- All managers are accessed via their `shared` singleton, making them easy to mock in tests if needed.
- The ECS pattern keeps logic localized to components; the state machine orchestrates high‑level flow.
- For rapid iteration, you can edit component behavior (e.g., adjust `MovementComponent.speed`) and re‑run the simulator without rebuilding the entire app.
